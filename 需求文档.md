AI助手-虚幻引擎插件需求文档
版本：2.0
更新日期：2026年4月18日

1. 背景与目标
1.1 背景
本插件为虚幻引擎编辑器集成AI助手功能，旨在通过大语言模型辅助用户分析蓝图、指导功能实现。AI助手通过调用插件提供的本地工具获取工程上下文、蓝图节点信息，并基于实时数据给出分析或开发建议。

1.2 核心目标
消除交互摩擦：AI应主动、静默地获取必要上下文，避免反复追问用户。

提升分析准确性：AI必须能精确理解蓝图执行流与数据流，不再混淆连线语义。

引导式开发体验：AI需具备分步指导能力，包括检查用户操作结果、复用已有资产、面向可视化界面的自然表述。

数据实时性保障：AI每次分析前必须通过工具重新拉取最新蓝图状态，杜绝缓存误判。

2. 现状问题分析
2.1 蓝图分析流程缺陷
问题场景	当前表现	根本原因
用户未打开蓝图	AI调用资产列表后逐条列举，并反复试探子文件夹（如“看看ThirdPerson有没有…”），对话冗长低效。	资产列表返回扁平路径，无层级结构。
用户未打开工程	AI可能尝试获取上下文或返回模糊错误，未明确提示用户打开工程。	工具调用前置条件校验缺失。
蓝图节点信息冗余	当前返回节点数据量过大，但AI实际只需连接拓扑与关键属性。	工具未做信息筛选，包含大量无用字段。
连线解析错误	用户连线：BeginPlay → MoveTo（执行），GetLocation → MoveTo.Location（数据）。
AI错误理解为：BeginPlay → GetPlayerCharacter → GetLocation → MoveTo 的线性执行流。	返回的JSON中未区分执行流与数据流，且无结构化连线描述。
2.2 资产目录视图不直观
当前 获取资产列表 工具返回扁平化的资产路径列表，AI无法一次性掌握 /Content 下的完整层级结构，导致频繁试探性调用工具，严重影响对话效率与智能感。

2.3 蓝图数据实时性无强制约束
插件虽提供实时查询接口，但AI在对话历史中可能依赖之前返回的旧数据回答用户问题，造成“蓝图已修改但AI仍基于旧图分析”的错误。

2.4 数据格式问题
当前返回的蓝图节点标题包含中文乱码（如 "è·³è·ƒ"），AI无法正确识别节点功能，导致分析失败。

节点信息缺失 引脚列表 与 连线关系，AI仅能获取节点类型与标题，无法理解蓝图逻辑。

3. 期望行为模式
3.1 静默环境感知
当用户发起与蓝图/功能实现相关的提问时，AI首先静默调用获取项目目录树工具，获得 /Content 完整层级结构。

AI不在对话中显式列出文件树，而是以自然语言确认环境。例如：

“我看到您当前打开的项目是 MyProject，其中已有 ThirdPersonCharacter 蓝图，我们可以基于它开始。”

3.2 引导式开发流程
AI需支持分步指导 + 中途验证 + 灵活复用的交互模式。标准流程示例如下：

阶段一：功能规划与资产复用
用户：“我想做一个敌人AI，会追逐玩家。”
AI（内部动作）：调用 get_content_tree 获取目录树。
AI（回复）：
“我看到您的项目中有一个 ThirdPersonCharacter 蓝图，已具备基础移动能力。建议如下操作：

在 Content/Blueprints 下复制 ThirdPersonCharacter，命名为 BP_Enemy。

打开 BP_Enemy，将其事件图表中的节点全部删除（组件保留）。

完成后告诉我，我会检查并指导添加追逐逻辑。”

阶段二：状态检查与纠偏
用户：“好了，节点删完了。”
AI（内部动作）：调用 get_blueprint_info 获取当前打开蓝图（BP_Enemy）的节点列表。
AI（回复）：

若图表已清空：
“确认事件图表已清空。现在添加以下节点：Event Tick、AI MoveTo……（详细步骤）”

若仍有残留节点：
“我注意到事件图表中仍存在 Event BeginPlay 节点，请将其删除后再继续。”

阶段三：错误排查
用户：“连线后运行报错，说无法移动。”
AI（内部动作）：调用 get_blueprint_info 获取当前完整节点与连线信息。
AI（回复）：
“根据当前蓝图，AI MoveTo 节点的 Pawn 输入引脚未连接。请将 Self 引用连到该引脚。”

3.3 资产复用策略
AI应根据目录树主动识别可复用的蓝图（如角色、控制器、行为树），并给出具体的复制/继承/引用建议。

指导命名时提供明确示例（如 BP_Enemy、BT_EnemyBehavior、EQS_FindPlayer）。

3.4 面向可视化界面的回答表述
AI内部处理JSON数据，但对用户的描述必须完全使用蓝图可视化界面的术语，严禁提及JSON结构、节点ID、引脚ID等内部标识。

示例对比：

错误表述	正确表述
“在JSON的nodes数组中找到BeginPlay，将其exec引脚连接到MoveTo的execute引脚。”	“从 Event BeginPlay 节点的执行输出引脚拖出连线，连接到 AI MoveTo 节点的执行输入引脚。”
“将GetActorLocation的ReturnValue引脚数据连接到Destination引脚。”	“将 Get Actor Location 节点的 Return Value 引脚连接到 AI MoveTo 节点的 Destination 引脚。”
4. 技术规范与接口要求
4.1 蓝图信息查询接口（get_blueprint_info）
调用时机
AI每次需要了解蓝图当前状态时，必须重新调用此接口，禁止使用历史对话中缓存的数据。

返回数据结构要求
返回 JSON 必须包含以下最小信息集合，并确保 UTF-8 编码正确，无乱码。

{

字段说明
字段	类型	必须	说明
nodes[].id	string	✅	节点唯一标识，用于连线引用
nodes[].type	string	✅	节点类名（如 K2Node_CallFunction）
nodes[].functionName	string	✅	函数/事件名（引擎内部名称）
nodes[].displayTitle	string	✅	用户可见的节点标题（UTF-8）
nodes[].isPure	boolean	✅	是否为纯函数（无执行引脚）
pins[].pinId	string	✅	引脚唯一标识
pins[].direction	enum	✅	"Input" / "Output"
pins[].category	enum	✅	"exec"（执行流）/ "data"（数据流）/ "object" 等
pins[].type	string	✅	引脚数据类型（如 exec, Vector, Actor）
pins[].defaultValue	string	❌	未连接时的默认值（可选）
links[]	array	✅	连线列表，至少包含一条时提供
links[].linkType	enum	✅	"exec" 或 "data"，明确区分连线性质
重要：category 和 linkType 是AI正确理解蓝图逻辑的核心字段，必须准确提供。

4.2 资产目录树查询接口（get_content_tree）
调用时机
用户提出涉及工程结构或蓝图创建的提问时，AI首先调用此接口获取全局视图。

返回数据结构要求
返回 /Content 目录下的完整递归树结构。

json
{
  "projectName": "MyProject",
  "contentTree": [
    {
      "name": "ThirdPerson",
      "type": "folder",
      "children": [
        {
          "name": "Blueprints",
          "type": "folder",
          "children": [
            {
              "name": "BP_ThirdPersonCharacter",
              "type": "blueprint",
              "path": "/Game/ThirdPerson/Blueprints/BP_ThirdPersonCharacter",
              "parentClass": "Character"
            }
          ]
        }
      ]
    }
  ]
}
字段说明
字段	说明
type	节点类型："folder"、"blueprint"、"map"、"behaviorTree"、"blackboard" 等
path	资产完整路径（引擎内部路径格式）
parentClass	仅蓝图类型提供，表示父类
该结构使AI能够一次调用即掌握全部资产分布，无需反复试探子目录。

4.3 通用要求
编码：所有字符串字段必须为合法的 UTF-8 编码，禁止出现乱码。

实时性：接口查询的数据必须反映当前编辑器中的实时状态，不得缓存。

错误处理：当用户未打开任何蓝图或未打开工程时，接口应返回明确的错误码与提示信息，供AI向用户友好表达。

5. AI系统提示词约束（概要）
以下内容应体现在AI助手的系统指令中：

工具调用策略：

用户提问后，首先静默调用 get_content_tree，在内部建立工程结构认知。

涉及蓝图状态（检查、报错、下一步指导）时，必须先调用 get_blueprint_info 获取最新信息。

严禁使用对话历史中缓存的蓝图数据回答问题。

蓝图解析规则：

必须严格区分 category: "exec" 的执行流连线与 category: "data" 的数据流连线。

执行流决定节点执行顺序，数据流决定参数传递，二者不可混淆。

分析蓝图逻辑时必须基于 links 数组重建拓扑关系。

交互行为规范：

不得在对话中显式列出返回的目录树或 JSON 数据。

回复必须使用蓝图可视化界面的自然语言，避免提及内部标识符。

指导用户操作时分步进行，并在关键步骤后主动检查用户蓝图状态。

复用与适配：

根据目录树主动推荐可复用的现有资产。

若用户项目缺少必要资产（如AI控制器），应指导用户创建并说明原因。

6. 约束与注意事项
类别	约束项	要求
数据	中文乱码	必须杜绝，所有字符串使用 UTF-8 编码输出。
数据	信息冗余	节点信息仅保留分析所需字段（见 4.1），避免传输无用属性。
交互	静默感知	AI 调用目录树接口后不得将原始数据呈现给用户。
交互	实时性	AI 每次分析蓝图前必须重新调用查询接口。
交互	术语统一	AI 对用户的表述必须符合虚幻引擎蓝图界面的通用术语。
功能	前置校验	工具应能识别“未打开工程”“未打开蓝图”等状态，并返回清晰错误码。
功能	执行/数据流区分	JSON 中必须通过 category 与 linkType 明确标识。
